# Задача: добавить режим квазистатического расчета в инженерное веб-приложение для стержневых конструкций

## Контекст проекта

Есть веб-система для работы со стержневыми 2D-конструкциями.

Сейчас приложение работает **только в статическом режиме**:

- пользователь собирает 2D-схему;
- задает узлы, элементы, нагрузки, закрепления;
- запускает расчет;
- получает итоговые результаты по конструкции;
- может добавлять дефекты;
- может пересчитывать влияние дефектов;
- получает AI-прогноз риска по элементам;
- есть отдельный модуль управления жизненным циклом ML-модели:
  - ведение датасета,
  - запуск/остановка обучения,
  - мониторинг метрик и логов,
  - выбор активной inference-модели,
  - экспорт PDF-отчета.

## Что нужно сделать

Нужно **дописать и частично переписать приложение**, чтобы кроме статического расчета появился режим **квазистатики**.

### Как понимать квазистатику в этом проекте

Квазистатика = это **последовательность статических пересчетов по шагам**.

То есть:

- расчет идет не в одной точке, а по шагам `step_1, step_2, ..., step_n`;
- на каждом шаге могут меняться:
  - нагрузки,
  - коэффициенты нагрузок,
  - параметры дефектов,
  - свойства элементов;
- на каждом шаге выполняется обычный статический расчет;
- результаты сохраняются по всей истории шагов;
- итогом является не один снимок состояния, а **траектория изменения состояния конструкции**.

### Что НЕ нужно делать

Сейчас **не** нужно реализовывать:

- полноценную динамику;
- уравнения движения;
- инерцию, массу, демпфирование;
- модальный анализ;
- ударные/вибрационные сценарии;
- сложную физику времени.

Нужен именно **пошаговый статический пересчет**.

---

# Цель изменений

Добавить в систему новый режим расчета: **"Квазистатика"**, который позволит:

1. задавать сценарий шагов;
2. менять параметры модели от шага к шагу;
3. запускать серию статических расчетов автоматически;
4. хранить результаты по каждому шагу;
5. показывать пользователю изменение:
   - перемещений,
   - усилий,
   - напряжений/деформаций (если они уже есть в системе),
   - влияния дефектов,
   - AI-риск-оценки по элементам;
6. отображать эволюцию риск-зон и тепловой карты;
7. включать результаты квазистатики в PDF-отчет.

---

# Что нужно проанализировать в кодовой базе

Сначала изучи текущую архитектуру и найди:

1. где описана модель конструкции:
   - узлы,
   - элементы,
   - материалы,
   - закрепления,
   - нагрузки,
   - дефекты;

2. где находится текущий статический расчет:
   - входные данные,
   - solver,
   - формат результатов;

3. где находится API:
   - endpoints для расчета,
   - схемы запросов/ответов,
   - фоновые задачи, если есть;

4. где расположен UI:
   - экран редактирования схемы,
   - запуск расчета,
   - отображение результатов,
   - риск-карта / heatmap;

5. где формируется PDF-отчет;

6. как сейчас встроен AI-модуль прогнозирования риска:
   - откуда берутся признаки,
   - в какой момент вызывается inference,
   - как сохраняются и отображаются результаты.

# Что нужно изменить

## 1. Ввести новый тип расчета

Добавить новый режим:

- `static`
- `quasi_static`

Если в системе уже есть enum / calculation type / analysis type — расширить его.

## 2. Добавить сущность сценария квазистатики

Нужна новая структура данных, например:

- `QuasiStaticScenario`
- `QuasiStaticStep`

Каждый шаг должен уметь хранить:

- номер шага;
- название шага;
- описание шага;
- коэффициент/масштаб нагрузок;
- изменения по дефектам;
- изменения по свойствам элементов;
- необязательные пользовательские параметры;
- порядок выполнения.

Пример логики:

- шаг 1: базовая нагрузка 0.25;
- шаг 2: базовая нагрузка 0.50;
- шаг 3: базовая нагрузка 0.75;
- шаг 4: базовая нагрузка 1.00;
- шаг 5: тот же уровень нагрузки + увеличение дефекта в выбранных элементах.

## 3. Реализовать движок пошагового расчета

Нужно сделать orchestration-слой, который:

1. берет исходную модель;
2. применяет изменения шага;
3. запускает существующий статический solver;
4. сохраняет результат шага;
5. повторяет для всех шагов сценария;
6. формирует агрегированный результат сценария.

Важно:

- по возможности **переиспользовать существующий статический solver**;
- не дублировать его без необходимости;
- новую логику строить как надстройку над текущим статическим расчетом.

## 4. Сохранение результатов по шагам

Нужно хранить:

- результаты каждого шага отдельно;
- агрегированные результаты сценария;
- связь между сценарием и конструкцией/проектом;
- связь с AI-риск-оценкой на шаге.

Для каждого шага желательно иметь:

- перемещения;
- внутренние усилия;
- напряжения/деформации, если уже поддерживаются;
- список дефектов в состоянии этого шага;
- AI-оценку риска;
- heatmap/risk-zones;
- статус шага (`pending`, `running`, `done`, `failed`).

## 5. Интеграция с AI-оценкой риска

После расчета каждого шага нужно иметь возможность:

- либо сразу запускать inference,
- либо запускать inference по флагу.

AI-риск должен считаться **на каждом шаге**, чтобы можно было видеть:

- как меняется риск элемента от шага к шагу;
- какие элементы становятся критичными;
- ранжирование элементов по риску на шаге;
- общую эволюцию risk heatmap.

## 6. Изменения в UI

Нужно добавить UI для режима квазистатики.

### На экране постановки задачи:

- переключатель режима: `Статика / Квазистатика`;
- редактор сценария шагов;
- возможность:
  - добавить шаг,
  - удалить шаг,
  - изменить порядок шагов,
  - настроить коэффициенты нагрузок,
  - задать изменения дефектов,
  - задать изменения параметров элементов.

### На экране результатов:

- список/таймлайн шагов;
- переключение между шагами;
- отображение результатов выбранного шага;
- графики/таблицы изменения показателей по шагам;
- обновляемая тепловая карта риск-зон;
- ранжирование элементов по риску на выбранном шаге;
- желательно режим проигрывания шагов как последовательности.

## 7. API и контракты

Нужно обновить backend API:

- создание/редактирование сценария;
- запуск квазистатического расчета;
- получение списка шагов;
- получение результата конкретного шага;
- получение агрегированной сводки по сценарию.

Если API уже существует для статического расчета — не ломать совместимость без необходимости.
По возможности:

- сохранить старые контракты;
- новые возможности добавить через новые поля или новые endpoints.

## 8. PDF-отчет

Нужно расширить экспорт PDF так, чтобы отчет по квазистатике включал:

- общую информацию о сценарии;
- таблицу шагов;
- ключевые результаты по каждому шагу;
- графики изменения параметров по шагам;
- AI-риск по шагам;
- итоговый список наиболее рискованных элементов;
- финальную heatmap / summary risk zones.

# Требования к реализации

## Архитектурные требования

- Максимально переиспользовать существующий статический расчет.
- Не ломать текущий режим статического расчета.
- Сделать расширение через понятные абстракции.
- Разделить:
  - доменную модель,
  - orchestration шага,
  - solver,
  - AI inference,
  - UI-представление результатов.
- Обеспечить расширяемость на будущее, чтобы потом можно было добавить:
  - зависимость параметров от шага по формулам,
  - импорт сценария,
  - более сложные временные сценарии.

## Нефункциональные требования

- Обратная совместимость со старым статическим режимом.
- Прозрачная обработка ошибок по шагам.
- Логирование выполнения сценария.
- Возможность перезапуска расчета сценария.
- Предсказуемая структура данных для фронта.
- Отсутствие дублирования логики там, где можно переиспользовать существующий код.

# Критерии готовности

Функциональность считается готовой, если:

- пользователь может выбрать режим `Квазистатика`;
- пользователь может создать сценарий из нескольких шагов;
- система выполняет статический расчет на каждом шаге;
- результаты доступны по каждому шагу;
- AI-риск обновляется по шагам;
- heatmap/risk-зоны можно смотреть по шагам;
- PDF-отчет поддерживает сценарий квазистатики;
- старый статический сценарий не сломан.
