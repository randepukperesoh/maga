# Саммари: как в проекте обучается нейросеть (1 лист A4)

## Назначение обучения
В текущем backend обучение нужно для получения коэффициентов линейной risk-модели, которая оценивает вероятность дефекта в каждом стержне конструкции. Обучение не решает задачу классической «глубокой» нейросети с множеством слоёв; по сути это калибруемая логистическая модель с инженерными признаками, интегрированная в pipeline расчёта и инференса.

## Источник данных
Обучение использует записи из `training_dataset` (таблица `training_dataset_samples`). Каждая запись содержит:
- геометрию конструкции (`nodes`, `rods`),
- нагрузки и закрепления,
- список дефектов (если размечены),
- опционально сенсорные данные,
- текстовую метку (`label`).

На этапе подготовки данные разворачиваются в обучающие строки «по стержням»: один инженерный кейс даёт сразу несколько объектов обучения (по числу стержней). Благодаря этому из ограниченного числа сценариев формируется больше наблюдений.

## Формирование признаков
Для каждого стержня вычисляются базовые признаки:
1. `bias = 1.0` (свободный член модели).
2. `length/slenderness` — относительная стройность стержня.
3. `area_factor` — производная от площади сечения (фактор «ослабления»).
4. `load_factor` — относительная интенсивность нагрузки в узлах стержня.
5. `defect_feature` — признак наличия/плотности дефектов по разметке.

Именно этот вектор используется в тренировочном модуле (`training_runtime.py`) для оценки вероятности дефекта через логистическую функцию.

## Разделение на train/validation
В проекте применяется групповой временной сплит (`group_temporal`):
- сначала данные группируются по `scenario_id`/`name`,
- затем более «поздние» группы уходят в validation,
- дополнительно контролируется, чтобы в train и validation были оба класса.

Это снижает риск слишком оптимистичной оценки, когда одинаковые или почти одинаковые примеры попадают одновременно в train и validation.

## Процедура обучения
Обучение выполняется как оптимизация логистической регрессии:
- оптимизируются веса признаков (`w_length`, `w_area`, `w_load`, `w_prior`) и `bias`,
- используется градиентный шаг по эпохам,
- применяются регуляризации: L2 и ограничение дрейфа весов относительно базовой модели,
- учитываются class weights для дисбаланса классов.

Если данных недостаточно, система переключается в fallback-режим (эвристическое обновление весов), чтобы сервис оставался работоспособным.

## Калибровка и порог решения
После базового обучения применяется:
1. Platt calibration (`a`, `b`) для калибровки вероятностей.
2. Подбор `decision_threshold` по validation (ориентир на баланс F1/accuracy/долю положительных).

Это важно, потому что в эксплуатационном режиме нужны не только «сырые» вероятности, но и стабильный порог для принятия решения о дефекте.

## Выходные артефакты
Результат обучения сохраняется в двух формах:
- runtime-состояние в БД (список моделей, активная inference-модель, история),
- checkpoint-файл `risk_model.json` с коэффициентами и метаданными.

В артефакте фиксируются:
- версия модели,
- коэффициенты,
- параметры обучения,
- размеры train/validation,
- метрики (`log_loss`, `accuracy`, `precision`, `recall`, `f1`, `balanced_accuracy`),
- итоговый порог решения.

## Как это используется в инференсе
В инференсе backend берёт активную модель из runtime-состояния. Далее:
- формирует признаки для каждого стержня,
- считает вероятность дефекта,
- ранжирует стержни по риску,
- возвращает `risk_by_rod`, `top_risky_rods`, оценку позиции дефекта и heatmap.

Таким образом, обучение в проекте — это не «offline отчёт», а встроенный механизм обновления production-коэффициентов для API `/predict-defect`.

## Практический вывод
Качество модели сильнее всего зависит от репрезентативности `training_dataset`: разнообразия схем, реалистичности нагрузок, корректности дефектной разметки и сенсорных сценариев. При бедном или слишком однотипном датасете модель быстро переобучается и даёт неустойчивые оценки. При хорошем покрытии кейсов — даёт полезный ранжированный риск и удобный эксплуатационный сигнал для диагностики.
