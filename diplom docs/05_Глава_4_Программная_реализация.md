# 4. Программная реализация

## 4.1 Назначение программного комплекса

Программный комплекс предназначен для автоматизации задач локализации дефектов в стержневых конструкциях на основе нейросетевой модели. Система реализована как веб-приложение, что обеспечивает доступность для инженеров без установки специализированного desktop-ПО.

Функции системы:
- прием и валидация входных данных мониторинга;
- запуск пайплайна предобработки и инференса;
- визуализация координат дефекта и аналитических показателей;
- хранение результатов и формирование отчетов.

## 4.2 Архитектура системы

Архитектура построена по клиент-серверной модели и включает четыре логических уровня:
1. Уровень представления (`Frontend`).
2. Прикладной уровень (`Backend API`).
3. Уровень аналитики (`ML service`).
4. Уровень хранения (`Data layer`).

Основные принципы архитектуры:
- слабая связанность компонентов;
- четкие API-контракты;

- журналирование и трассируемость операций.

## 4.3 Backend: логика обработки и инференса

### 4.3.1 Функциональные модули backend

Backend включает:
- модуль расчета стержневой схемы (статический solver + пошаговый quasi-static orchestration);
- модуль дефектов (CRUD дефектов, пересчет с учетом дефектов);
- модуль AI-инференса риска по стержням и сегментам (`risk_heatmap`);
- модуль управления ML-жизненным циклом (история обучения, логи, активная inference-модель, датасет);
- модуль отчетности (генерация PDF-отчета с таблицами, графиками и схемой конструкции).

### 4.3.2 Последовательность обработки запроса

1. Прием запроса и первичная валидация.
2. Проверка структуры входного набора.
3. Извлечение и нормализация признаков.
4. Передача тензоров в ML-модуль.
5. Получение предсказания координат.
6. Расчет дополнительных диагностических показателей.
7. Запись результата и возврат ответа клиенту.

### 4.3.3 Требования к надежности

- обработка некорректных входных форматов;
- тайм-ауты и повтор запросов к ML-модулю;
- идемпотентность операций записи;
- централизованное логирование с идентификатором транзакции.

## 4.4 Frontend: интерфейс инженерного пользователя

Интерфейс ориентирован на сценарии эксплуатации и должен обеспечивать минимальное когнитивное усилие при интерпретации результата.

Ключевые элементы:
- 2D-редактор узлов и стержней с интерактивным canvas;
- панель нагрузок, закреплений и параметров quasi-static шагов;
- панель результатов расчета и AI-оценки риска;
- визуализация риск-карты по стержням;
- модуль экспорта PDF-отчета.

Требования к UX:
- однозначное отображение единиц измерения;
- указание уровня достоверности прогноза;
- отсутствие «скрытых» критичных параметров;
- поддержка мобильного и desktop-режимов для просмотра результатов.

## 4.5 Взаимодействие компонентов

Взаимодействие организовано через REST API. Пример логической структуры ответа:
- `analysisType`: тип анализа (`quasi_static`);
- `quasiStaticSteps`: результаты по шагам (перемещения и напряжения);
- `stresses`: итоговые напряжения;
- `risk_by_rod` и `risk_heatmap`: риск-оценка по стержням и сегментам;
- `top_risky_rods`: ранжирование наиболее рискованных стержней.

Такой контракт обеспечивает:
- предсказуемую интеграцию frontend/backend;
- удобство трассировки и аудита;
- поддержку версионирования API.

## 4.6 Развертывание и сопровождение

Система поддерживает контейнеризованное развертывание. Эксплуатационный контур включает:
- сервис приложения;
- сервис модели;
- БД и файловое хранилище;
- reverse-proxy для маршрутизации;
- средства мониторинга доступности и ошибок.

Рекомендуемые практики сопровождения:
- версионирование модели и датасетов;
- автоматизированные smoke-тесты после деплоя;
- мониторинг задержек инференса;
- регламент безопасного обновления модели.

## 4.7 Информационная безопасность и доступ

Минимальные меры ИБ:
- разграничение ролей (оператор, эксперт, администратор);
- хранение журналов действий пользователя;
- защита API-токенов и секретов;
- контроль целостности конфигурации модели.

## 4.8 Выводы по главе

Реализована модульная программная архитектура, обеспечивающая интеграцию нейросетевого алгоритма в прикладной веб-контур. Разделение ответственности между frontend, backend и ML-сервисом обеспечивает масштабируемость, сопровождаемость и практическую пригодность решения.

## 4.17 Фактически реализованный контур приложения

В рамках практической реализации в приложении зафиксирован режим **quasi-static по умолчанию**. Пользовательский контур ориентирован на инженерную работу со схемой, дефектами и интерпретацией риска, без отдельного сценарного мастера на фронтенде.

Ключевые технические решения текущей версии:
- quasi-static режим принудительно активен в API и клиенте;
- используется встроенный default-сценарий шагов нагружения (0.25, 0.50, 0.75, 1.00), если пользователь не задал собственные шаги;
- результаты расчета и AI-инференса возвращаются в согласованном пошаговом формате;
- PDF-отчет формируется как расширенный инженерный документ: summary-карточки, графики, таблицы и схема конструкции.

В PDF-отчет включены:
- диаграмма распределения напряжений по стержням;
- тренд по квазистатическим шагам (max |stress| и max risk);
- таблицы `Top Risky Rods`, `Defect Table`, `Quasi-static Step Summary`;
- схема модели, где стержни отрисованы градиентом вероятности дефекта по данным `risk_heatmap`, а дефекты отмечены на геометрии с подписью локальной вероятности.

Таким образом, программная реализация доведена до состояния прикладного инженерного инструмента: от построения модели и расчета до интерпретируемой риск-визуализации и формализованного отчета.

## 4.9 Дополнительная проработка: API-контракты

Типовой набор REST-эндпоинтов:
- `POST /api/v1/prediction` — запуск диагностики;
- `GET /api/v1/prediction/{id}` — получение результата;
- `GET /api/v1/health` — проверка доступности;
- `GET /api/v1/models` — информация о версии модели.

Минимальная структура запроса на инференс:
- идентификатор объекта;
- набор сигналов/признаков;
- метаданные условий измерения.

Минимальная структура ответа:
- координаты дефекта;
- confidence;
- версия модели;
- диагностические предупреждения.

## 4.10 Дополнительная проработка: обработка ошибок

В производственном контуре необходимо разделять типы ошибок:
1. Ошибки валидации входа (`400`).
2. Ошибки авторизации (`401/403`).
3. Ошибки отсутствия ресурса (`404`).
4. Ошибки ML-сервиса (`502/503`).
5. Непредвиденные ошибки (`500`).

Для каждого типа должна быть:
- машиночитаемая кодировка;
- человекочитаемое описание;
- рекомендация по восстановлению.

## 4.11 Дополнительная проработка: производительность и масштабирование

Метрики производительности:
- медианная и 95-й перцентиль задержки ответа;
- пропускная способность (запросов/сек);
- доля ошибочных запросов;
- время холодного старта ML-сервиса.

Инструменты масштабирования:
- горизонтальное масштабирование backend-реплик;
- выделение inference-сервиса в отдельный пул;
- кэширование повторяющихся запросов;
- асинхронная очередь для длительных задач.

## 4.14 Детализация: e2e-сценарии использования

### Сценарий S1: штатная обработка

1. Пользователь загружает набор измерений.
2. Система валидирует входные данные.
3. Backend формирует признаки и запускает инференс.
4. Frontend отображает координаты дефекта и confidence.
5. Пользователь экспортирует отчет.

### Сценарий S2: неполные данные

1. Загружены данные с пропусками каналов.
2. Система выставляет `diagnostic_flag`.
3. Выдается результат с пониженной достоверностью или отказом.
4. Пользователь получает рекомендацию по повторному измерению.

### Сценарий S3: пакетная обработка

1. Пользователь отправляет серию файлов.
2. Backend помещает задачи в очередь.
3. Результаты постепенно возвращаются по ID задач.
4. Итоговый сводный отчет формируется автоматически.
